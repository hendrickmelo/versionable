## Part 1: Git-Tag Versioning with `hatch-vcs`

Use git tags as the single source of truth for version numbers, similar to how amix
derives versions from tags. `hatch-vcs` (built on `setuptools-scm`) reads the nearest
`v*` tag at **build time** and bakes the version into the package — no runtime git
dependency needed.

### Version format

| Scenario | Example |
| --- | --- |
| On a tag `v0.0.1` | `0.0.1` |
| 3 commits after `v0.0.1` | `0.0.2.dev3+gabc1234` |
| Dirty working tree on tag | `0.0.1+d20260406` |

### 1.1 Update `pyproject.toml`

```toml
[build-system]
requires = ["hatchling", "hatch-vcs"]
build-backend = "hatchling.build"

[project]
dynamic = ["version"]           # remove the hardcoded version = "..."

[tool.hatch.version]
source = "vcs"

[tool.hatch.build.hooks.vcs]
version-file = "src/versionable/_version.py"
```

This auto-generates `src/versionable/_version.py` at build time with:

```python
__version__ = "0.0.1"
```

### 1.2 Expose `__version__` in `__init__.py`

```python
from versionable._version import __version__
```

For editable installs (dev workflow), `_version.py` is generated by `pip install -e .`
or by running `hatch version`. Add `_version.py` to `.gitignore` since it's generated.

### 1.3 Dynamic Version in `docs/conf.py`

```python
from importlib.metadata import version
release = version("versionable")
```

Remove the hardcoded `release = "0.0.1"`.

### 1.4 `pixi.toml` Version

The `version` field in `pixi.toml` is workspace metadata only — it doesn't affect the
published package. Leave it as-is or remove it.

## Part 2: PyPI Publishing

### 2.1 Fix Metadata Gaps

Add to `[project.urls]` in `pyproject.toml`:

- `Documentation = "https://versionable.readthedocs.io"`
- `Issues = "https://github.com/hendrickmelo/versionable/issues"`

### 2.2 Create Publish Workflow

`.github/workflows/publish.yml` triggered on GitHub releases:

- Build sdist + wheel with `python -m build`
- Publish via `pypa/gh-action-pypi-publish` with trusted publisher (OIDC)
- Separate build and publish jobs, `pypi` GitHub environment for protection

### 2.3 Manual: PyPI Account Setup

1. Check name availability at https://pypi.org/project/versionable/ (fallback: `py-versionable`)
2. Configure Trusted Publishing on PyPI (repo = `hendrickmelo/versionable`, workflow = `publish.yml`, environment = `pypi`)
3. Create `pypi` GitHub environment in repo settings

### 2.4 Update Install Instructions

Replace git-based install in README with:

```bash
pip install versionable
pip install "versionable[hdf5]"  # with HDF5 support
```

Keep the git install as a dev alternative.

## Part 3: Read the Docs

### 3.1 Add `docs` Extra to `pyproject.toml`

```toml
[project.optional-dependencies]
docs = ["sphinx>=7", "myst-parser>=2", "furo"]
```

This lets RTD (and contributors) install doc deps without pixi.

### 3.2 Create `.readthedocs.yaml`

Use RTD's native Python build with pip:

```yaml
version: 2

build:
  os: ubuntu-24.04
  tools:
    python: "3.12"

sphinx:
  configuration: docs/conf.py

python:
  install:
    - method: pip
      path: .
      extra_requirements:
        - docs
```

### 3.3 Manual: RTD Setup

1. Sign in to https://readthedocs.org with GitHub
2. Import `hendrickmelo/versionable`
3. Verify build at `https://versionable.readthedocs.io`

## Risks

- **Package name:** `versionable` may be taken on PyPI -- check first
- **numpy dependency:** heavyweight but appropriate for target audience
- **Python 3.12+:** conscious choice, limits audience
- **`hatch-vcs` requires git history:** shallow clones (e.g., `actions/checkout` default) need `fetch-depth: 0` in CI workflows
